## 背景

`PLAN.md`作成時点（および`CURRENT_STATE.md`/`DB_MIGRATION_FINDINGS.md`の調査時点）では、「JV-Dataの払戻（HR）レコードはどのパーサーにもマッピングされておらず、馬連・ワイド・三連複等の確定払戻金データはリポジトリのどこにも存在しない」という前提で`BET-0`を最優先ブロッカーとしていた。

harness運用中（Loop1）にGeneratorが新規HRパーサー実装を行い、新テーブル`payouts`へのUPSERTを試みたところ、Evaluatorが「実テーブルのスキーマが想定と異なる」ことを検出。これを人間が直接DB確認したところ、**`fukurou_jvdl.payouts`テーブルが既に存在し、258,565行のデータが投入済み**であることが判明した。前提が誤っていたため、以下3点を再調査した。

---

## 調査1: 5賭式（単勝/複勝/馬連/ワイド/三連複）それぞれのデータ有無

**結論: 全て存在する。** `payouts.bet_type`の内訳（全258,565行）:

| bet_type | 行数 | 対応する賭式 |
|---|---|---|
| tansho | 21,676 | 単勝 ✓ |
| fukusho | 64,488 | 複勝 ✓ |
| umaren | 21,680 | 馬連 ✓ |
| wide | 65,023 | ワイド ✓ |
| sanrenpuku | 21,692 | 三連複 ✓ |
| wakuren | 20,519 | 枠連（`PLAN_INPUT.md`要件外だが既存） |
| umatan | 21,717 | 馬単（同上） |
| sanrentan | 21,770 | 三連単（同上） |

`PLAN_INPUT.md`の確定要件である5賭式は全てカバーされている。8賭式全てで`min(race_id)`/`max(race_id)`が完全に一致（`202001050601`〜`202604050912`）しており、特定賭式だけが欠落している箇所はない。

---

## 調査2: 「当たり馬券のみ存在し、外れ馬券（0円）は存在しない」仮説の検証

**結論: 仮説は正しい。ただしこれは欠損ではなく、JV-Data払戻レコード(HR)の正規の構造である。**

レースごとの行数分布を確認した結果:

| bet_type | 平均行数/レース | 分布 |
|---|---|---|
| tansho / umaren / sanrenpuku / wakuren / umatan / sanrentan | 1.00 | ほぼ全レースで1行（同着時のみ2行以上） |
| fukusho | 2.98 | 大半が3行（出走数<8等で2行、同着で4-5行のケースもあり） |
| wide | 3.00 | 大半が3行（同着で5-7行のケースもあり） |

これは実際のJRA確定払戻表の構造そのものである（単勝は1着のみ、複勝は3着まで、ワイドは3組のみが「払戻対象」であり、的中しなかった組番にはそもそも払戻が発生しないため記録自体が存在しない）。

**この仮説が意味すること（設計上の結論）:**
回収率計算の「投資側（分母）」を`payouts`テーブルから作る必要はない。分母は「本命×相手の組み合わせ選定ロジック（`BET-1`/`BET-2`/`BET-3`）が何点購入したか」から決まるものであり、`payouts`はそのうち「的中した組番がいくら払い戻したか」を引くための参照テーブルとして使えば十分である。選定した組み合わせが`payouts`に見つからない場合は「不的中（リターン0）」であり、データ欠損ではない。

**ただし区別が必要な点:** 上記は「個々の組み合わせが`payouts`に存在しない」場合の話であり、「そのレース・その賭式の行が`payouts`に1行も存在しない」場合（後述の調査3）は意味が異なる。前者は的中判定の結果（正常）、後者はデータ取り込みの欠損（異常、N/A対象）である。既存`tipster/backtest.py`の`_aggregate_picks`も単勝・複勝の的中判定をレース結果（`races_v2`等の確定着順）から行っており、的中なし=0円という同じ考え方を既に採用している。

---

## 調査3: データの欠損・網羅性

**結論: 直近データに重大な欠損がある。** `races_v2`（`is_jra=true`、JV-DataのJRA本場開催レース）と`payouts`を対応するレースキーで突合した結果:

### 3-1. 軽微・無視できる欠損（歴史的）
- 1989年4月〜5月: 425レース欠損（JV-Dataの確定払戻記録がそもそも遡れない古い時期と推測）
- 1993〜1998年の散発的な欠損（1〜4件/月、合計15件程度）

### 3-2. 軽微だが原因不明の欠損
- 2026年2月7日〜9日（jyo=05東京, 08京都, 10小倉）に42レース分が欠損。特定の開催日・場のみが丸ごと欠けており、原因（取り込み漏れか、配信側の欠落か）は本調査では特定できていない。

### 3-3. 重大な欠損（直近・現在進行中）

**`payouts`データの取り込みは2026年4月5日で停止している。** それ以降の確定済みレースの払戻データが一切存在しない。

| 年月 | JRA本場レース数 | `payouts`欠損数 | 欠損率 |
|---|---|---|---|
| 2026-01 | 276 | 0 | 0% |
| 2026-02 | 324 | 42 | 13%（3-2の欠損） |
| 2026-03 | 300 | 0 | 0% |
| 2026-04 | 264 | 216 | **82%**（4/6以降が全欠損） |
| 2026-05 | 336 | 336 | **100%** |
| 2026-06 | 192 | 192 | **100%**（21日まで開催済みの分を含む） |

**影響:** `PLAN_INPUT.md`の確定要件「検証データ: 2025年6月〜（直近まで）」を字義通り満たそうとすると、2026年4月6日〜現在（最大約2.5ヶ月分）の検証データで5賭式回収率検証ができない。これは`BET-0`が「ゼロから作る」のではなく「直近データを追いつかせる（バックフィル）」必要があることを意味する。

### 3-4. 補足: `race_id`のフォーマット不一致（実装上の注意点）

`payouts.race_id`（12桁、例: `202512200611`）と`races_v2.race_id`（16桁、例: `2024070447060310`）は構成が異なる:

- `payouts.race_id` = `年(4)+月日(4)+場コード(2)+race_num(2)`（開催回・日目を含まない）
- `races_v2.race_id` = `kaisai_year(4)+kaisai_monthday(4)+keibajo_code(2)+kaisai_kai(2)+kaisai_nichime(2)+race_num(2)`（開催回・日目を含む）

両者を突合する際は`races_v2.kaisai_year || kaisai_monthday || keibajo_code || race_num`という形で開催回・日目を除いたキーに変換する必要がある。単純な`race_id`同値比較ではJOINが失敗する（実際に検証中、誤った前提でJOINした際は28,649件中852件しかマッチせず、原因を特定した）。Generatorが`payouts`を参照する実装を書く際にこの変換を正しく行う必要がある。

## PLAN.mdへの反映方針

上記を踏まえ、`PLAN.md`の`BET-0`/`BET-3`/`G6`/`§1-1`/`§4`を以下の方向で修正する（Blocker基準`G1-G4, G5a, G5b`は変更しない）:

1. `BET-0`の目的を「HRレコードパーサー新規実装」から「既存`payouts`テーブルのスキーマに整合する形でのデータ取り込みコード整備 + 2026-04-06以降の欠損分バックフィル」に再定義する。
2. `BET-0`のDone条件に「既存`payouts`の258,565行を破壊しないこと（既存スキーマへの追加・適合であり、置き換えではない）」を明記する。
3. `BET-3`のDone条件に「`payouts`に組み合わせが存在しない＝不的中（リターン0）」と「そのレース・賭式が`payouts`に1行も存在しない＝データ欠損（N/A、集計除外）」を区別するロジックが必要であることを明記する。
4. `G6`（Blocker外、High）に「レース単位の欠損判定」の具体的な合格ラインを追記する。
5. `race_id`フォーマット変換の注意点を`BET-0`の設計制約として明記する。
