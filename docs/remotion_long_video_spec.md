# 【プロジェクト要件定義】YouTube横型動画（長尺）自動生成システム

## 1. プロジェクトの目的と基本方針
すでに稼働している競馬予測AI（4-submodel構成）の推論結果JSONを用いて、Remotion (React + Tailwind CSS) で16:9の横型動画を完全自動生成する。
「単調なデータ読み上げ動画」からの脱却を図り、視聴維持率を高める「テレビライクなリッチUI」と「メリハリのある構成」、そして「キャラクターの掛け合い」を実装する。
**究極のルール：「重賞は深く、平場は広く面白く」**

## 2. UI/UXデザインの絶対ルール（テレビ品質の追求）
安っぽい量産型動画を避けるため、以下のルールをコンポーネント設計の基準とする。

* **世界観とトーン＆マナー（ディープグリーン）:** 知的で落ち着いたトーンをベースとし、過度な和風やサイバー感は避ける。イメージカラーである深めの緑（例: `bg-emerald-950`, `bg-green-900`）を基調とし、座布団やフレームに適用してクリーンな質感を持たせる。
* **絵文字の完全禁止と独自アセット化:** OS依存の絵文字は使用しない。必ず独自のSVGアイコンで表現し、チャンネルの独自性を担保する。
* **テキストの「座布団」と「エッジ」:** テキストの直打ちは禁止。必ずTailwindを用いて半透明の背景（座布団）を敷く。また、CSSを用いて文字にフチ取りとシャドウをつけ、視認性を極限まで高める。
* **L字レイアウトの徹底:** 画面を左右にスプリットし、左側（約35%）を「キャラクター立ち絵＋字幕枠」、右側（約65%）を「データ枠」として視線を固定化する。
* **静止画面の排除:** 背景には`<AbsoluteFill>`を用いてループアニメーションを配置し、フリーズ感を排除する。

## 3. キャラクター設定と対話ロジック（ゆっくり実況スタイル）
動画は単なる読み上げではなく、2名のキャラクターによるテンポの良い「掛け合い（漫才）」形式で進行し、視聴者の離脱を防ぐ。

* **フクロウ博士（天才AI・主役）:**
  * **設定:** 一人称「わし」。データと物理法則（Estimated Maximum Potential）のみを信奉する冷徹な天才。自信満々で少し煽り気味だが愛嬌がある。語尾は「〜だホー」。
  * **役割:** 助手の浅はかなオッズ信仰をZ-scoreで論破し、真の推奨馬を提示する。
* **助手（ひよこ・視聴者の代弁者）:**
  * **設定:** 一人称「僕/オレ」。競馬新聞の印やオッズ、直近の着順にすぐ騙される典型的な競馬ファン。語尾は「〜っす」「〜じゃないすか！」。
  * **役割:** 視聴者が抱く疑念（「前走大敗してるのに買えるわけない！」等）を先回りして大げさに代弁するツッコミ役。
* **動的演出（フラグ）:** * セリフの文脈に合わせてLLMに `pose`（立ち絵の変更：default, pointing, shocked, begging等）や `camera_zoom`（助手へのズームイン等）の演出フラグを決定させる。

## 4. フォント戦略とシチュエーション別・テキスト演出
フォントの使い分けと、動画の展開（感情の起伏）に合わせた4つのテキスト演出コンポーネント（CSS装飾）を実装する。

* **用途別フォントの指定:**
  * **ベース（日本語）:** `Noto Sans JP` (ウェイト: Black/900) など、極太のゴシック体。
  * **データ（数字・英字）:** `Oswald` などのスタイリッシュな欧文フォント。将来的な数字のアニメーション化を考慮し `<DataValue>` コンポーネント等でラップする。
* **シチュエーション別演出コンポーネント:**
  1. **通常・解説モード（ベース）:** 白文字＋深い緑や黒のドロップシャドウ。落ち着いた座布団と組み合わせて高い視認性を確保する。
  2. **アラートモード（危険な人気馬）:** 人気馬のマイナスデータを突きつける緊迫感を演出。黄色や赤などの警戒色を用いたフチ取り（`-webkit-text-stroke`）を適用する。
  3. **スパイスモード（特注・血統穴馬）:** メインとは違う独自視点での穴馬紹介。深緑のベースに映えるゴールドや紫などのアクセントカラーのテキストシャドウを適用する。
  4. **激アツモード（鉄板馬・本命結論）：** 最大のフックとなる本命発表や突出したZ-scoreに適用する**パチンコ風リッチテキスト（疑似3D）**。黄金グラデーション（`-webkit-background-clip: text`）＋極太輪郭線＋多重 `text-shadow` を用いた圧倒的な立体感をCSSのみで再現する。

## 5. 開催スケジュール別：出し分け（ルーティング）戦略
インプレッション分散を回避し、検索ボリュームを最大化する。
* **パターンA（2場/1重賞）:** 平場パック（金曜21時） / 重賞特化動画（金曜19時）
* **パターンB（2場/2重賞）:** 平場パック（土曜21時） / ダブル重賞スペシャル（土曜19時）
* **パターンC/D（3場開催）:** レース数が増えても尺は伸ばさずテンポでカバーし、パターンA/Bと同様に2本立てで運用。

## 6. 動画テンプレート構成（動的タイムライン設計）
※固定尺（絶対時間）での定義は廃止。総尺は「各コンポーネントのTTS（音声）長」および「レース数」からデータ駆動で動的算出（Audio-driven duration）する。

### A: 重賞特化型動画（目安：8〜10分）
* **【シーン1：フック】:** 危険な人気馬の結論先出し（掛け合い / 尺：音声依存）
* **【シーン2：危険な人気馬】:** Z-scoreのマイナス要素による論破（掛け合い / 尺：音声依存）
* **【シーン3：特注スパイス】:** 血統・展開（事前シミュレーション）の特異要素からの穴馬紹介（掛け合い / 尺：音声依存） **※LLMへの厳秘事項：調教要素はAIモデルV3アップデート時に解禁予定のため、現在は絶対に言及しないこと。**
* **【シーン4：本命解剖】:** レーダーチャート・SHAP値を用いた完全証明（掛け合い / 尺：音声依存）
* **【シーン5：買い目】:** バックテストに基づく期待値の高い買い目提示（尺：固定15秒等）

### B: 平場・特別戦パック動画（目安：8〜10分）
* **【シーン1：OP】:** 鉄板馬と穴馬の存在予告（掛け合い / 尺：音声依存）
* **【シーン2：サクサク予想】:** EV平凡レースの高速処理（尺：1レースあたり約15秒固定 × レース数）
* **【シーン3：鉄板馬コーナー】:** 基礎能力突出馬の解説（掛け合い / 尺：音声依存）
* **【シーン4：特注スパイス】:** 血統・展開の起爆剤馬の紹介（掛け合い / 尺：音声依存） **※LLMへの厳秘事項：同上、調教への言及は一切禁止。**
* **【シーン5：ED】:** まとめと買い目提示（掛け合い / 尺：音声依存）

## 7. 起爆剤コンテンツ：抽出ロジックとコーナーの棲み分け
動画テンプレートの意図に合わせて、抽出ロジックの適用範囲を厳格に定義する。

1. **【平場パック専用】鉄板馬コーナー ＝「絶対能力のバケモノ」**
   * **抽出:** `ability_v2` Z-score「+2.5以上」かつ2位と一定以上の差。
   * **台本:** 助手が「堅すぎる」と疑う中、博士が「AIが弾き出した基礎能力スコアがバグっている」「物理法則として勝つ確率が最も高い」と断言。
2. **【重賞・平場 共通】特注スパイス（穴馬）コーナー ＝「AIだけが気づいているバグ馬」**
   * **抽出:** `ability_v2` は平凡だが、`pace_v2` または `pedigree_v1` が「+2.0以上」。
   * **台本:** 助手が「前走大敗してるオッズ50倍なんて買えない」とツッコミを入れ、博士が「事前展開シミュレーションが超前傾ラップを検知した」とロジカルに煽る。
3. **【重賞特化専用】危険な人気馬 ＝「危険な『高実績』馬（ハリボテ）」**
   * **抽出:** 当日上位人気必至だが、`course_v2` や `pace_v2` がマイナス。
   * **台本:** 助手が「この1番人気は鉄板」と信じ込むのに対し、博士が「実績に騙されるな」「直線の物理シミュレーションでは足が止まる」と一刀両断する。

## 8. 技術アーキテクチャ (Remotion & LLM連携)
Remotionにおける「動的尺」と「リッチUIレンダリング」の破綻を防ぐための絶対要件。

* **構造化プロンプト（JSON）による台本制御:**
  * LLM（Claude等）には単一テキストではなく、発言ごとのオブジェクト配列（`dialogue`）を出力させる。
  * JSON構造例: `[ { "speaker": "助手", "text": "〜っすか!?", "telop": "要約字幕", "pose": "shocked", "camera_zoom": "assistant_full" }, ... ]`
* **音声合成（TTS）と尺同期フロー（Audio-driven duration）:**
  * バックエンドはLLMが生成した台本を音声API（VOICEVOX等）に渡し、音声ファイル（mp3/wav）を生成する。
  * Remotionに渡すJSONには、各シーンの `audio_url` と同時に `audio_duration_ms`（音声の秒数メタデータ）を必ず含める。
  * Remotion側は `<Sequence durationInFrames={Math.ceil((audio_duration_ms / 1000) * 30) + マージン}>` のように設定し、テキスト量に依存する映像尺の自動追従を実現する。
* **データ駆動型グラフィック描画（SVGレーダー・SHAPチャート）:**
  * グラフやチャートは、バックエンドから静止画（PNG）として生成・出力してはいけない。
  * 構成要素の生データ（Z-score, SHAP値など）をJSONでRemotionに渡し、フロントエンド側で `Recharts` 等のReact描画ライブラリと `spring()` アニメーション関数を用いて、動的かつ滑らかに描画する。
