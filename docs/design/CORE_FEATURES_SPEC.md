## コアバリュー宣言

本 SaaS の最重要差別化軸は以下の2点である。

1. **「AIのブラックボックス化排除」** — なぜそのランク・スコアなのか、すべてにファクトの根拠を持つ
2. **「生データによる納得感の提供」** — ユーザーが自分の目でデータを確認し、自分の判断として結論を出せる UI

これを実現するための専用ページとして、以下の2機能を定義する。

## 1. 🏁 レースレベル（次走追跡）検証ページ

### 1.1 目的と位置づけ

**目的:**  
Pro馬柱の `RaceScore`（S/A/B/C ランク）の根拠を、対戦馬の次走成績という「ファクト」で完全に証明する専用ページ。  
ユーザーが「なぜこの過去レースがハイレベルと判定されたのか」を 100% 自分で検証・納得できる。

**呼び出し元:**  
Pro馬柱の `RaceScore` バッジ（各過去走セル）をクリック/タップしたとき、このページへ遷移する。  
バッジには `race_id`（過去走の race_id）を持たせ、URL パラメータとして渡す。

**URL:**  
```
/race-level/:past_race_id
```

例: `/race-level/2026042605020212`

---

### 1.2 ページ構成（上から順）

#### セクション A: レースサマリーヘッダー

対象過去レースの基本情報を表示する。

| 表示項目 | データソース | 備考 |
|---------|------------|------|
| レース名 | `races.race_name_hondai` | 例「東京4R ダート1400m」 |
| 開催日 | `races.race_date` | |
| 競馬場・距離・馬場種別 | `races.keibajo_code` / `distance` / `track_code` | |
| 馬場状態 | `races.shiba_baba_code` / `dirt_baba_code` | 良/稍重/重/不良 |
| 出走頭数 | `races.syusso_tosu` | |
| グレード | `races.grade_code` → 表示ラベル変換 | G1/G2/.../未勝利 |
| **RaceScore** | 算出済み | ランクバッジ（S/A/B/C）と内訳（タイム指数/メンバー/クラス） |

`track_condition_warning: true` の場合はヘッダー内に以下の警告バナーを表示する:  
```
⚠ 馬場状態の混在あり — 同日比較対象レースで馬場状態が異なるため、タイム指数の精度が低下している可能性があります。
```

---

#### セクション B: 動的フィルタリングパネル

ユーザーが分析条件を変更できるフィルター UI。フィルター変更は即時（React state）でセクション C・D を再計算する。

**フィルター項目:**

| フィルター名 | UI 種別 | デフォルト値 | 説明 |
|-----------|--------|-----------|------|
| 対象着順（上限） | スライダー or セレクトボックス（1〜出走頭数） | `5` | 「何着以内の馬を集計対象とするか」 |
| 着差（上限・秒） | スライダー（0.0〜5.0秒 / 0.1刻み） | `5.0`（制限なし） | 「勝ち馬との着差が X 秒以内の馬のみ対象」。0.5 に設定すると僅差の激戦馬のみが抽出される |
| 自馬を除外 | チェックボックス | オフ | 予想対象馬が当該レースに出走していた場合に除外するオプション |

フィルター変更時の挙動:
- フィルターに該当する馬が 0 頭になった場合: 「条件に合致する馬がいません。フィルターを緩和してください。」と表示
- サンプル数（`n`）が 3 未満の場合: 「サンプル数が少なく（n=X）、統計的信頼性が低い可能性があります。」と注記

---

#### セクション C: 次走成績 分布マトリクス

フィルター条件に合致した馬の「次走成績」をマトリクス形式で可視化する。

**マトリクス構造:**

```
行（縦）: 当該過去レースでの着順グループ
列（横）: 次走の結果カテゴリ
```

**行の定義（当該過去レースでの着順グループ）:**

| 行ラベル | 条件 |
|--------|------|
| 1着    | `this_rank = 1` |
| 2着    | `this_rank = 2` |
| 3着    | `this_rank = 3` |
| 4〜5着 | `this_rank IN (4, 5)` |
| （フィルターで除外された着順は非表示） | |

**列の定義（次走結果カテゴリ）:**

| 列ラベル | 条件 |
|--------|------|
| 1着    | `next_race_rank = 1` |
| 2着    | `next_race_rank = 2` |
| 3着    | `next_race_rank = 3` |
| 4着以下 | `next_race_rank >= 4` |
| 未出走  | `next_race_rank IS NULL`（次走データなし） |

**セル内の表示:**
- 該当馬の頭数（例: `3頭`）
- 割合（例: `(50%)`）

**マトリクス下部のサマリー行:**
- 「対象 X 頭中、次走3着以内: Y 頭（Z%）」を太字で強調
- この値が `member_level_score` の計算根拠そのものであることを説明テキストで明示する:
  ```
  ※ この好走率（Z%）が「メンバーレベルスコア XX pt」の計算根拠です
  ```

**カラーハイライト:**
- 次走1〜3着セル: 背景を薄い緑（好走）
- 次走4着以下セル: 背景を薄い赤/灰（凡走）
- 未出走セル: 背景を薄い黄（データなし）

**追加クラス別集計（折りたたみ可能な拡張セクション）:**

次走のクラス（G1/G2/G3/OP/3勝/2勝/1勝/未勝利/新馬）ごとにグループ化し、  
「このレースから G1/G2 に格上げされた馬が何頭いるか」を表示する。

```
次走クラス別内訳:
  G1 出走 → 1頭（2着）
  G2 出走 → 2頭（1着・3着）
  G3 出走 → 1頭（4着）
  OP  出走 → 2頭（1着・5着）
  ...
```

#### セクション D: 個別ファクト一覧

フィルター条件に合致した全馬の次走成績を、1頭1レコードのカードまたはテーブル行として表示する。  
ユーザーが自分の目で1頭ずつ確認できることが重要。

**1レコードに含まれる情報:**

| フィールド | 表示内容 | 例 |
|----------|--------|---|
| 当該レース着順 | `this_rank` 着 | `2着` |
| 着差 | `this_margin` 秒差 | `(+0.3秒)` |
| 馬名 | `horse_name`（DBから取得） | `ロードカナロア産駒A` |
| 次走レース名 | `next_race_name`（DBから取得） | `東京スポーツ杯2歳S (G2)` |
| 次走開催日 | `next_race_date` | `2026-06-15` |
| 次走着順 | `next_race_rank` | `1着` |
| 次走頭数 | `next_head_count` | `/14頭` |
| テキストサマリー | 自動生成 | 「**2着**だった本馬は、次走 **東京スポーツ杯2歳S(G2)** に昇級し **1着** を収めた」 |

**テキストサマリーの自動生成ルール（Python/TypeScript ロジックベース、LLM 不使用）:**

```
パターン1（次走1着）:
「[this_rank]着だった[horse_name]は、次走[next_race_name]([grade])で1着を収めた」

パターン2（次走2〜3着）:
「[this_rank]着だった[horse_name]は、次走[next_race_name]で[next_rank]着と好走した」

パターン3（次走4着以下）:
「[this_rank]着だった[horse_name]は、次走[next_race_name]で[next_rank]着だった」

パターン4（未出走）:
「[this_rank]着だった[horse_name]の次走データは未取得（未出走または取得期間外）」
```

**ソート順:** デフォルトは当該レース着順（1着→2着→...）の昇順。  
ユーザーが「次走着順」でソート可能なオプションボタンを提供する。

## 2. 🗺️ 展開ストーリー・不利予測ページ

### 2.1 目的と位置づけ

**目的:**  
AI の隊列予想（逃げ/先行/差し/追込のポジショニングマップ）を出力するだけでなく、  
「なぜその並びになり、誰が物理的に不利を受けるのか」を馬番・枠番・テン速度の具体的な数値から  
ユーザーが読めるストーリーとして提示する。

「AIが追い込みを指名した根拠は、外枠から無理に先行すると大きなロスが生じるという物理的制約を加味したからだ」という説明を、ユーザーが直感的に理解できる形で提供する。

**呼び出し元:**  
レース詳細画面（`RaceDetailView`）の Standard タブ内、現行の「AI 隊列予想パネル（`PositioningMapPanel`）」の隣または下部に「展開ストーリーを見る →」リンクを設置する。

**URL:**
```
/race-story/:race_id
```

例: `/race-story/202606060511`

---

### 2.2 ページ構成（上から順）

#### セクション A: レースヘッダー

レース詳細と展開予想ラベルを表示する。既存の `RaceHeaderPanel` と同一情報で構成する。

| 表示項目 | 備考 |
|---------|------|
| レース名・距離・馬場種別 | |
| 展開予想ラベル | ハイペース / 平均 / スロー（色付きバッジ） |
| コース形態 | 右回り/左回り・芝/ダート（将来拡張で自動取得） |

---

#### セクション B: 枠順 × テン速度 ビジュアライゼーション

**最重要セクション。** 馬の物理的な位置と速度を視覚化し、展開ストーリーの根拠を提示する。

**表示構造（横並び: 内枠 → 外枠）:**

```
内枠                                                     外枠
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ 枠 1 │ 枠 2 │ 枠 3 │ 枠 4 │ 枠 5 │ 枠 6 │ 枠 7 │ 枠 8 │
│      │      │      │      │      │      │      │      │
│ 馬番 │ 馬番 │ 馬番 │ 馬番 │ 馬番 │ 馬番 │ 馬番 │ 馬番 │
│ 馬名 │ 馬名 │ 馬名 │ 馬名 │ 馬名 │ 馬名 │ 馬名 │ 馬名 │
│      │      │      │      │      │      │      │      │
│ テン │ テン │ テン │ テン │ テン │ テン │ テン │ テン │
│ 指数 │ 指数 │ 指数 │ 指数 │ 指数 │ 指数 │ 指数 │ 指数 │
│ バー │ バー │ バー │ バー │ バー │ バー │ バー │ バー │
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
```

同一枠に複数馬がいる場合は縦に並べる（16頭立てで1枠に2頭が入ることを前提とする）。

**テン指数バーの仕様:**
- データソース: 既存の `avg_first_corner_norm_5` 特徴量を反転して 0〜100 に正規化した `ten_index`
  - `ten_index` が高い = テンが速い（逃げ・先行）
  - `ten_index` が低い = テンが遅い（差し・追込）
- バーは枠カラー（現行の `FRAME_CLS` 定義）で着色する
- **数値ラベル:** バー上部に `ten_index` の数値を表示（例: `87`）
- **脚質ラベル:** `ten_index` からの自動判定ラベルを表示:
  - `>= 75`: 🏇 逃げ（赤系テキスト）
  - `55〜74`: 先行（オレンジテキスト）
  - `35〜54`: 差し（グレーテキスト）
  - `< 35`: 追込（紫テキスト）

**視覚的強調ルール:**
- テン指数が最高の馬（ハナを争う最有力候補）に「🥇 最速」バッジを付与
- テン指数上位 2〜3 番手の馬に「速」マーカーを付与
- 同一枠内で特に速い馬を強調表示（複数馬の枠で内外のテン速度差が大きいケース）

#### セクション C: 不利リスクカード一覧

物理的ロジックから「不利を受けやすい馬」を自動判定し、1頭1カードで提示する。  
判定はすべてルールベース（LLM 不使用）で行い、根拠を数値で示す。

**不利パターン定義と判定ロジック:**

| パターン ID | パターン名 | 判定条件（擬似コード） | 生成ストーリーテンプレート |
|-----------|-----------|---------------------|------------------------|
| `PACE_CUT` | 前をカット | `自馬のten_index >= 60` かつ `直外の馬のten_index > 自馬のten_index` かつ `直外の馬.umaban == 自馬.umaban + 1` | 「**{馬名}** は枠{枠番}から先行を図る予定だが、すぐ外の{直外馬名}（テン{直外テン}）がさらに速いため、カットされて砂（またはキック）を被るリスクがある。ポジション低下→直線での消耗が懸念される。」 |
| `HANA_ARASOI` | ハナ争い激化 | テン指数70以上の馬が3頭以上 | 「逃げ候補が{N}頭（{馬名リスト}）存在し、ハナ争いが激化するリスクが高い。ハイペース化の可能性が大きく、先行馬全体への消耗が懸念される。」 |
| `OUTER_SENSEN` | 外枠の前争い | テン指数70以上の馬が5枠以外の外枠に集中（5〜8枠に2頭以上）している | 「外枠に速いテンの馬（{馬名リスト}）が集中。これらの馬が内に切れ込む動きによってペースが上がりやすく、前有利な展開が崩れる可能性がある。」 |
| `INNER_SLOW` | 内枠の先行馬が不利 | テン指数が低い馬（差し・追込）が内枠（1〜2枠）に入った | 「**{馬名}** は差し脚質（テン{テン}）だが内枠{枠番}に入った。出遅れた場合に内で詰まるリスクがあり、外に持ち出すロスが大きくなりやすい。」 |
| `OUTSIDE_DISADVANTAGE` | 外枠の追込馬 | テン指数が低い馬（追込）が外枠（7〜8枠）に入った | 「**{馬名}** は追込脚質（テン{テン}）で外枠{枠番}。位置が後ろになりやすく、外を回る距離ロスが加算される可能性がある。ただし直線が長いコースでは問題になりにくい。」 |
| `PACE_HARMONY_MISS` | 展開不一致 | 既存の `pace_harmony_pre < 0.3` | 「**{馬名}** の得意な展開（{得意パターン}）と、今回の予測展開（{展開ラベル}）が大きく乖離している。逆境での走りとなる可能性が高い。」 |

**カード表示項目:**

```
┌─────────────────────────────────────────────┐
│ ⚠️ [パターン名]               リスク: ★★★☆☆ │
│                                              │
│ 馬番 [N] [枠色] [馬名]                       │
│                                              │
│ [ストーリーテキスト（上記テンプレートを充填）]  │
│                                              │
│ 根拠データ:                                   │
│   自馬のテン指数: XX                          │
│   直外馬のテン指数: YY                        │
│   pace_harmony_pre: Z.ZZ                    │
└─────────────────────────────────────────────┘
```

**リスク星評価（★☆☆☆☆ 〜 ★★★★★）の算出ルール:**

| 星数 | 条件 |
|-----|------|
| ★5 | `HANA_ARASOI` かつ `PACE_CUT` が同時発生 |
| ★4 | `HANA_ARASOI` 単独、または `PACE_CUT` かつ `PACE_HARMONY_MISS` |
| ★3 | `PACE_CUT` 単独、または `OUTER_SENSEN` |
| ★2 | `INNER_SLOW` または `OUTSIDE_DISADVANTAGE` |
| ★1 | `PACE_HARMONY_MISS` 単独 |

複数パターンが同一馬に発生した場合は、最も高い星数を採用し、全パターンをカード内に列挙する。  
不利リスクが特定されなかった馬は「特段の不利なし」と記載したグレーカードを1枚表示する（存在感なし）。

#### セクション D: 展開ストーリー全体サマリー

ページの末尾に「このレース全体の展開ストーリー」を自動生成された段落テキストとして表示する。

**生成ロジック:**

以下の条件分岐に従い、複数の文を連結する（ルールベース、LLM 不使用）。

```python
story_parts = []

# 1. ペース予測の宣言
if field_pace >= 0.55:
    story_parts.append("ハイペース濃厚な一戦。")
elif field_pace >= 0.35:
    story_parts.append("平均的なペースが予想される。")
else:
    story_parts.append("スローペースに落ち着きやすい一戦。")

# 2. ハナ候補の言及
if len(hana_candidates) == 1:
    story_parts.append(f"{hana_candidates[0].name}（{hana_candidates[0].ten}）がハナを主張する見込み。")
elif len(hana_candidates) >= 2:
    names = "・".join(c.name for c in hana_candidates[:3])
    story_parts.append(f"{names}らがハナを争い、序盤から激しい先行争いになる可能性がある。")

# 3. 外枠有力先行馬への言及
if len(outer_fast_horses) >= 2:
    story_parts.append(f"外枠から{outer_fast_horses[0].name}ら速いテンの馬が内に切れ込む動きで、さらにペースが上がるリスクがある。")

# 4. 最高リスク馬への言及
if high_risk_horses:
    for h in high_risk_horses[:2]:
        story_parts.append(f"{h.name}は{h.risk_description}。")

# 5. 恩恵を受ける馬への言及
if field_pace >= 0.55 and sashi_oikomi_horses:
    names = "・".join(h.name for h in sashi_oikomi_horses[:3])
    story_parts.append(f"ハイペースで前が崩れれば、{names}など後方勢に展開が向く可能性がある。")
elif field_pace < 0.35 and nige_senko_horses:
    names = "・".join(h.name for h in nige_senko_horses[:3])
    story_parts.append(f"スローで先行馬有利な展開では、{names}がそのまま押し切るシナリオに注意。")

return "".join(story_parts)
```

**表示例（生成テキスト）:**

> ハイペース濃厚な一戦。ドンレパルス（テン87）がハナを主張する見込み。外枠からオレデイイ・グレイテスらが内に切れ込む動きで、さらにペースが上がるリスクがある。モーニングサンは枠5から先行を図る予定だが、直外のグレイテス（テン91）にカットされて砂を被るリスクがある。ハイペースで前が崩れれば、ゴールドハーバー・フィアスら後方勢に展開が向く可能性がある。

### 2.3 必要な新規 API / データ拡張

#### 既存データの活用（新規 SQL 不要）

以下のデータは既存のエンドポイント `GET /api/v2/races/{race_id}` のレスポンスに含まれている:

| データ | フィールド | 既実装 |
|------|---------|-------|
| テン指数 | `horses[].extra.ten_index` | ✅ |
| 展開予想ラベル | `race_info.pace_prediction` | ✅ |
| AI 隊列（逃げ/先行/差し/追込） | `race_info.positioning_map` | ✅ |
| 馬番・枠番 | `horses[].umaban`, `.wakuban` | ✅ |

#### 新規追加が必要なデータ

| データ | 取得方法 | 優先度 |
|------|---------|-------|
| `pace_harmony_pre` | `_build_features()` の出力に含まれているが API レスポンスには未公開。`RaceDetailHorse` に追加が必要 | 高 |
| `predicted_position_norm` | 同上（`_build_features()` 出力には存在） | 高 |

#### 新規エンドポイント（不要）

展開ストーリーの生成に必要なデータは既存の `GET /api/v2/races/{race_id}` レスポンスを拡張するだけで取得可能。  
新規エンドポイントは作成しない。  

ただし、ストーリー生成ロジック（セクション C・D）は:
- **バックエンド生成案**: FastAPI 側の `GET /api/v2/races/{race_id}` レスポンスに `race_story` フィールドを追加
- **フロントエンド生成案**: レスポンスデータを受け取ったフロントエンドが JavaScript でストーリーを生成（ロジック管理が容易）

**推奨: フロントエンド生成案を採用する。**  
理由: ストーリーロジックはドメイン知識ベースのルール文字列連結であり、API を通す必要性がない。  
チューニングをフロントエンドのみで完結できるため、反復開発が速い。

## 4. 設計上の絶対制約

1. **LLM 不使用** — ストーリー生成・不利判定はすべてルールベース。API コスト・遅延・ハルシネーションリスクを排除する
2. **既存 API の破壊的変更禁止** — 新フィールドは `| null` で追加のみ。既存フィールドは削除・型変更しない
3. **データリーク防止の継承** — 両ページとも「当走の事後データ（実際の着順・ペース）」は参照しない。過去走データの次走結果は参照してよい（それがファクト）
4. **フォールバック必須** — データ不足（head_count = 1 など）や `ten_index = null` の馬がいても、UI がクラッシュせず graceful degradation すること
5. **モバイルファースト** — Tailwind の `md:` ブレークポイントでのみ広いレイアウトを展開する。モバイルでは縦スクロールを基本とする
