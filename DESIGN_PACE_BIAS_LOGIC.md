# 予想ロジック設計書（展開 × トラックバイアス編）

> 本書は、ユーザー（競馬歴5〜6年）の頭の中にある予想ロジックを言語化し、
> 条件化・AI学習・説明レイヤーに落とし込むための設計書である。
> **これは下書きであり、ユーザー確認後に確定する。**
> 確認時の観点：「自分の競馬知識が正確に言語化されているか」「抜け漏れ・誤解はないか」

## 0. この設計書の位置づけ

- 対象：本命選定の「幹」となる、最も重視している評価軸
- ユーザーの言葉：「これを見てほぼほぼ馬券を買っていると言っても過言ではない」
- 既存の「全条件を横並びでクリア数カウント」方式の限界を超えるための再設計
- 既存システム（conditions_v2.py 等）は壊さず、新エンジンとして並走させる

## 1. 基本思想：評価は「2階建て」

ユーザーの予想は、固定的な競馬場構造と変動的な当日状況の2層で構成される。

### 1階：競馬場の構造的特徴（固定・前日に分かる）
その競馬場・コースが本質的に持つ性質。集計で算出可能。

- **先行有利度**：基本的に競馬は先行有利（道中で隊列が固まり動きにくいため）。
  ただし競馬場により程度が異なる。
  - 大箱コース（例：東京）→ ゆったり回れて直線が長い → ポジション取りの重要度が低い
    → 強い馬が素直に来る → 人気サイドで決まりやすい
  - 小回り・急カーブ（例：中京）→ 内枠・先行が極端に有利、外に膨れると不利
- **内枠／外枠有利度**：競馬場・コース固有（例：中京の1枠の成績が異様に良い）
- **上がり勝負になりやすいか**：先行有利の逆で「上がりが使える馬でないと差せない」競馬場
- **荒れやすさ**：人気薄が来やすい競馬場か（例：最近の東京は荒れ傾向。
  他場で癖が出て実力を発揮できなかった馬が、東京の長い直線で覚醒するパターン）

### 2階：当日のトラックバイアス（変動・当日 or 直近から推定）
その日その瞬間の馬場の偏り。

- **前残り度**：先行有利 ←→ 差し有利
- **枠バイアス**：内枠有利 ←→ 外枠有利
- **上がり傾向**：上がり何番手までが実際に来ているか

### 評価の最終形
```
最終評価 = 馬の脚質・隊列適性
         × 1階（競馬場の構造的特徴）
         × 2階（当日のトラックバイアス）
```

ユーザー確認ポイント:
- この2階建ての理解で合っているか？（確認済み：合っている）
- 「トラックバイアスが先にあり、それと当日レースを照合して、隊列・上がりがどう使えるかを見る」
  という順序で合っているか（確認済み：合っている）

## 2. 隊列予想ロジック（誰がどの位置で運ぶか）

レースの並びを予想し、各馬が「運びやすいか」を判定する。

### 2-1. 脚質判定
- 各馬の過去レースの**4角（第4コーナー）の位置取り**から脚質を判定
  - 前方寄り → 先行馬
  - 後方寄り → 差し・追込馬
- 「先行したい馬」＝過去レースで前方に位置していた馬

### 2-2. テン速度の指標
- **テン3F（最初の600m）のタイム**を主指標とする
- 補足：前半ラップ全体も参考にできる
  - ただし前半ラップは「競り合い」「外から被せた」等の影響を含むため、
    その馬の特徴を読む材料として扱う

### 2-3. 距離変化による補正
- **距離延長**（例：1200m → 2000m）→ 前につけやすくなる
  - 理由：距離が延びるとペースが緩むため
  - 短距離で先行していた馬が距離延長 → 前で運べる可能性が高い
    （ユーザー注：一般的競馬知識として扱う。偏見の可能性も自認しているが採用してよい）
  - 1200mのテン速度をそのまま使うのではなく、距離が延びた分緩める補正をかける
    （「1200mでは速くて後方だったが、2000mならスタート緩いので前につけられる」等）
- **減量騎手への乗り替わり** → 前に行きがち（要考慮）

### 2-4. 隊列の攻防（できれば考慮）
- 内枠の先行馬が前を譲らない vs 外から被せてくる馬
- テン速度をさらに上げて競り合うか、譲って内で運ぶか
- この攻防がペースを左右する

### 2-5. 隊列予想の出力
- 各馬の「想定通過順位（想定ポジション）」
- レース全体の「想定ペース」（先行馬が何頭いるか）
  - 先行馬1頭のみ → 楽逃げ・前残り想定
  - 先行馬3頭以上 → ハイペース・差し有利想定

## 3. トラックバイアス判定ロジック

### 3-1. 競馬場固有特徴（1階・集計ベース）
全競馬場・全コースについて、過去データから以下を集計：
- 枠番別成績（内枠有利／外枠有利）
- 脚質別成績（先行有利／差し有利）
- 上がり順位別の好走率（上がり何番手まで来るか）
- 人気別決着傾向（堅い／荒れる）

これは前日でも分かる固定情報。

### 3-2. 当日バイアス（2階・推定ベース）
当日の馬場の偏りを推定する。場合分けあり：

#### 開幕週の場合
- 芝が綺麗、先行が残りやすい（一般則）
- → 先行有利・内枠有利と仮定

#### 2週目以降の場合
- 馬場が荒れてくる → 外差しが効くようになる傾向
- 前日予想時：**先週同曜日のレース結果**から推定
  - 朝一レースから最終レースまでの変化を見る
    （徐々に上がりが使える馬が来るようになっていれば差し有利化）
  - 変化がなければ内枠・芝の内側がまだ生きていると判断

#### 当日確定時
- **当日の既走レース結果**からバイアスを確定
- メインレース（11R等）の予想は、その前のレースまでの結果を見て判定
- 土曜の早いレース（1〜3R）はバイアス情報がなく予想が成り立たない
  → 「捨てレース」扱い。無理に推奨しない（プロの予想家も同様）

### 3-3. 運用上の時間軸
```
【土曜の予想（前日＝金曜夜〜土曜朝にSNS投稿）】
  - 開幕週 → 一般則（先行・内有利）
  - 2週目以降 → 先週日曜の結果からバイアス推定

【日曜の予想】
  - 土曜のレース結果からバイアス判定（精度高い）
  - 重賞・大レースは日曜が多く、当日型が最も活きる

【さらに上を目指す運用】
  - 当日メインレース前までのレース結果を見てリアルタイム更新／通知
    （「前走までの傾向が変わってきたのでこちらが良い」とSNS通知）
```

## 4. 馬の評価軸

### 4-1. 自在タイプ（最も評価したい・堅い）
- 過去に「**先行して勝ったレース**」と「**差して勝ったレース**」の**両方**がある馬
- 展開を見て自分でレースを選べる馬 → どんな展開でも対応できる
- 「先行がきつそうなら位置を取り、スタミナ勝負なら上がりを使う」
- ユーザーの呼称：「自在タイプ」（スタミナタイプに近い、好位から差せる馬）
- **判定にはキャリアが必要**（先行勝ち・差し勝ち両方の実績が溜まるまで判定不可）
  - キャリア浅い馬（2〜3戦）では判定不能 → 別ルートで評価（下記4-4）

### 4-2. 隠れた末脚（市場が見落とす・妙味）
- **上がり3Fの「実質順位」**で評価する
- 問題意識：ネット競馬等では上がり1〜3位までしか色（ハイライト）がつかない
  - 上がり最速から0.1秒遅いだけで全体4番手の馬は色がつかない → みんな見落とす
  - しかし実質は上位の脚を使えている → 妙味がある
- → 「色はつかないが実質上位（4〜5番手等）の上がりを使えた馬」を拾う
- **リーク注意**：当該レースの上がりは事後情報。過去走の上がり実質順位を使う

### 4-3. スピード適性（高速馬場トレンド対応）
- 最近の高速馬場決着トレンドを踏まえ、スピード適性のある馬を評価
- 上がりが使える＝道中それなりの位置で運び、直線で勝負をかけられる
- 先行が多いレースで上がりを使える馬は特に狙い目

### 4-4. キャリアの浅い馬の扱い
- 自在タイプ判定（4-1）は実績が必要なため適用できない
- 基本方針(c)：その馬の「現時点で得意な脚質」が、今日の展開・バイアスに合えば評価する
- 自在タイプは「さらに堅い」というボーナス的位置づけ
  （自在でなくても、得意脚質×今日の展開・バイアスが合えば本命候補になる）

### 4-5. 前走のレース運び
- 前走でちゃんとレースを運べていたか
- 4〜6着でも、展開が向かなかっただけなら本命候補になり得る
- 買い方の例：本命から3連複、または単勝1点＋他で回収（ユーザーの最近の好み）

## 5. 実装の3層構造

ユーザーの要望「展開・バイアスは複雑なのでAIに学ばせたいが、説明はちゃんとしたい」を
満たすため、3層に分離する。

### 第1層：数値化（ルールベース）
人間が定義できる計算は条件・ルールで数値化し、AIの入力特徴量を作る。
- 脚質判定（4角位置から）
- テン3F取得・距離補正・減量騎手補正
- 隊列予想（誰が前に行くか、想定ポジション、想定ペース）
- トラックバイアス指標（競馬場固有特徴 ＋ 先週/当日結果からの当日バイアス）
- 上がり実質順位（過去走、リーク防止）
- 自在タイプ判定（先行勝ち・差し勝ち両方の有無）

### 第2層：組み合わせ判断（AI）
第1層の特徴量を使い、「この馬は今日の展開・バイアスで好走しやすいか」を学習。
- 複雑な相互作用（隊列 × バイアス × 脚質 × スピード適性）を AI が捉える
- 人間が言語化しきれない部分も拾う
- 穴馬AIで採用したサブモデル方式・残差学習の思想を流用可能
- **リーク対策を最優先**（過去に sire_feature_store でルックアヘッドの事故あり）
  - ターゲットエンコーディングは予測日より前のデータのみ（PIT化）
  - 時系列分割（学習／検証を厳格に分ける）

### 第3層：説明（条件ベース）
AIが高評価した馬を、人間の言葉で「なぜか」を後付け説明。
- 「先行馬が1頭で前残り想定。この馬は先行力があり、今日の内有利バイアスに合致」
- AIスコアは裏方。対外的な説明は条件・日本語で行う（JRA-VAN規約とも整合）
- SNSでの「言い切り」に使える（ユーザーの強み）

```
数値化（ルール） → 組み合わせ判断（AI） → 説明（条件・日本語）
   第1層              第2層                 第3層
```

## 6. 拡張スロット（後から要素を足せる設計）

- 今回作るのは「幹」＝展開 × バイアス
- ユーザーには他に見る要素が**10個近く**ある（当日出馬表を見てスキップする要素含む）
- これらは「枝」として、後から1つずつ差し込める設計にする
- 穴馬AIのサブモデル方式をそのまま活用：
  - 各要素を独立したサブモデル or 条件として追加
  - メタモデルで幹のスコアと統合
  - 既存（幹）を壊さず差し込める構造を最初から用意しておく

```
【幹】展開 × バイアス エンジン ──┐
【枝1】要素1 ───────────────────┤
【枝2】要素2 ───────────────────┼─→ メタモデルで統合 → 最終推奨
  ...                            │
【枝10】要素10 ──────────────────┘
```

## 8. 実装ステップ（提案）

1. **本設計書の確認・確定**（ユーザーが明日レビュー）
2. **データ実現可能性の調査**（第7章の表をClaude Codeが埋める）
3. **第1層（数値化）の実装** — 取得可能なデータから順次
   - 脚質判定、テン3F、隊列予想、バイアス指標、上がり実質順位、自在タイプ
4. **第1層の検証** — 数値化が妥当か（隊列予想が実際の隊列と合うか等）
5. **第2層（AI）の実装・学習** — リーク対策を最優先、時系列分割厳守
6. **第2層の検証** — ホールドアウト、探索と検証の分離
7. **第3層（説明）の実装** — AIスコア → 条件・日本語への変換
8. **既存システムとの統合** — 幹として並走、拡張スロット用意
9. **運用テスト** — 土曜/日曜の場合分け、当日更新の動作確認
